Phase 1 — Get your clearance
Beat00

Are you a registered agent?

Confirm Git works & sign your name to it.

Before any mission, HQ checks you exist. Ask the terminal if it even knows what Git is:

terminal
$ git --version

No answer? You're a civilian — install Git from git-scm.com/downloads, then come back.

Now sign the roster. Every piece of intel you send gets stamped with your name — do this once, ever:

terminal
$ git config --global user.name "Your Name"
$ git config --global user.email "you@example.com"
✓ Cleared
Beat01

The secret handshake

Set up an SSH key so HQ trusts you — no passwords.

HQ won't trust a password shouted across a room. It trusts a secret handshake only you can do — a key with two halves. One half you keep locked away forever; one half you hand to HQ.

Forge the pair (press Enter to accept the default; add a passphrase — you're a spy, protect it):

terminal
$ ssh-keygen -t ed25519 -C "you@example.com"

Reveal only the public half and copy the whole line:

terminal
$ cat ~/.ssh/id_ed25519.pub

Walk it into HQ: GitHub → Settings → SSH and GPG keys → New SSH key → paste → save. Then rehearse the handshake:

terminal
$ ssh -T git@github.com
# "Hi you! You've successfully authenticated" = HQ knows your face
✓ Cleared
Phase 2 — Prep the intel (don't leak!)
Beat02

Reserve an empty locker

Make a new repo on GitHub — and leave it EMPTY.

Your intel needs somewhere to land. Go to github.com/new and set up a locker (a "repository"):

① Name it — e.g. sec-automation.  ② Pick Private (you're a spy, not a billboard).  ③ Leave every "Add a README / .gitignore / license" box UNCHECKED.

Rookie ambush

If HQ pre-fills the locker, its history won't match yours and your first delivery gets rejected at the door. Empty locker = clean delivery. Keep this page open — you'll grab its address in Beat 07.

✓ Cleared
Beat03

The "never leaves the building" list

Create a .gitignore so secrets can't escape.

This is the beat that separates pros from casualties. Before you pack anything, write the list of things that must never leave the safehouse — this file is called .gitignore. Paste this in your project folder:

terminal · run inside your project folder
$ cat > .gitignore << 'EOF'
# CLASSIFIED — these never leave the building
.env
*.pem
*.key
secrets.yaml
credentials.json
# Python debris
__pycache__/
*.pyc
.venv/
# Noise
*.log
.DS_Store
EOF

Tap the black bar to see the kind of thing you're protecting: API_KEY=sk_live_9f3aX… — if that ever reaches HQ in the open, the mission is blown.

✓ Cleared
Beat04

Arm the recorder & sweep the room

git init, then read git status like your clearance depends on it.

Turn on the mission recorder, then shine a light on everything it's about to capture:

terminal
$ git init
$ git status

Read every line. See a .env, a key, or a config with a password that slipped past your list? Freeze. Add it to .gitignore and sweep again. Ten seconds here beats a week of damage control.

✓ Cleared

Phase 3 — Pack, seal, and send
Beat05

Pack the crate

git add — load your cleared files, then double-check.

Everything passed the sweep? Load it into the crate, then peek inside one more time before you seal it — trust nothing, verify twice:

terminal
$ git add .
$ git status
# only what's listed here will ship — last check for anything classified
✓ Cleared
Beat06

Seal the crate

git commit — freeze a snapshot with a clear label.

Seal it and label what's inside. This snapshot is frozen forever — that's a commit:

terminal
$ git commit -m "Initial drop: automation scripts and gitignore"

Good labels are short and say what changed. Future-you, reading this at 2am, will be grateful.

✓ Cleared
Beat07

Point the courier at HQ

git remote add — tell Git which locker to run to.

Grab the SSH address from your GitHub repo page and give your courier its route. We nickname the destination origin:

terminal · swap in your own address
$ git remote add origin git@github.com:username/sec-automation.git
$ git remote -v
# confirms the courier knows the route

No SSH? Use the HTTPS address (https://github.com/username/repo.git) and paste a Personal Access Token when asked for a password.

✓ Cleared
Beat08

Fly the right flag

git branch -M main — run under the modern banner.

Every modern operation flies under the banner main. Raise it:

terminal
$ git branch -M main
✓ Cleared
Beat09

Send it 🎯

git push — dispatch the crate to HQ.

The moment of the mission. Dispatch the crate:

terminal
$ git push -u origin main

The -u tells the courier "remember this route," so next time you just type git push. Refresh your GitHub page — your intel is at HQ.

The standing orders

Every mission after the first

Once the route is set, the whole op is just three moves. This is what you'll actually do every day.

daily loop
$ git add .
$ git commit -m "what changed"
$ git push
# and to grab updates from HQ:
$ git pull
Operation II — Parallel Ops

Branches, Pull Requests & Merges

You've shipped solo. Now you'll work like a real team: run a job on a private side-track, get it reviewed, then fold it into the official record. main is that official record — never experiment on it directly.

Field Ops
0 / 5

Tap a stage to trace it on the map. Your work rides the side-track the whole way — main stays safe until the merge.

Op IIA

Split off a covert cell

git switch -c — make a private branch so main stays untouched.

Never run experiments on the official record. Peel off your own isolated track and step onto it in one move:

terminal
$ git switch -c feature/add-json-output

Older guides use git checkout -b feature/add-json-output — same thing. You're now working on a copy; anything you break here can't hurt main.

✓ Cleared
Op IIB

Work in secret, report to HQ

Commit as usual, then push the branch up to GitHub.

On your branch you use the exact same loop as before — nothing new to learn. Do your work, then send the branch to HQ:

terminal
$ git add .
$ git commit -m "Add JSON output format"
$ git push -u origin feature/add-json-output

The branch now exists at HQ — but it's not part of the official main record yet. That's the next beat.

✓ Cleared
Op IIC

Request clearance to merge

Open a Pull Request on GitHub and get it reviewed.

This part happens on the GitHub website, not the terminal. After you push, GitHub shows a "Compare & pull request" button. Click it to open a Pull Request (PR) — a formal "please review my work and fold it into main."

A teammate (or you) reads the PR's diff — the exact lines that changed — and approves or asks for edits. Reading that diff is code review, one of the most valuable habits you can build. When it's approved, GitHub shows a green Merge button.

✓ Cleared
Op IID

Fold it into the record

Merge the PR, then sync your local main.

Click Merge on GitHub. Your side-track work is now part of the official main record at HQ. Pull that update down to your own machine so you're in sync:

terminal
$ git switch main
$ git pull
$ git branch -d feature/add-json-output   # tidy up the finished branch
✓ Cleared
Op IIE

When two agents clash

Resolve a merge conflict — the thing everyone fears, defused.

Sometimes two people changed the same lines. Git can't guess who's right, so it stops and asks you to decide. Don't panic — this is routine. Bring main's latest into your branch to surface any clash early:

terminal
$ git switch feature/add-json-output
$ git merge main
# if it says "CONFLICT", open the file — you'll see markers like below

Inside the file, Git marks the clash. Your job: pick the right version (or blend them), then delete all three marker lines.

what a conflict looks like
<<<<<<< HEAD        your version of the line
your version of the line
=======
the incoming version
>>>>>>> main        the incoming version

Once the file reads the way you want and the markers are gone, seal the reconciliation:

terminal
$ git add .
$ git commit

The field toolkit

Gadgets every agent carries

Not a mission — just the everyday tools you'll reach for constantly. Tap Copy on any of them.

git clone

Drop into an existing HQ

Someone already built the repo? Copy the whole thing to your machine — working area, vault, and remote link, set up in one move. It's the reverse of git init.

terminal
$ git clone git@github.com:username/their-repo.git
git log

Read the mission log

Every sealed snapshot — who, when, and why. The --oneline --graph view is the readable one, and shows how branches split and merged.

terminal
$ git log --oneline --graph --all
git diff

Compare the intel

See the exact lines you changed before you pack or seal. Plain git diff shows edits still on your Desk; add --staged to inspect what's already in the Crate.

terminal
$ git diff
$ git diff --staged
git restore

Rewind a mistake

Grabbed the wrong file? Pull it back out of the Crate. Wrecked a file and want the last saved version back? Restore it. Neither touches history — that's revert, over in Code Red.

terminal
$ git restore --staged <file>   # take it out of the crate
$ git restore <file>            # discard edits, back to last commit
Bonus★

Contribute to another agency

fork → PR — how you help on a repo you don't own (a.k.a. open source).

You can't push directly to someone else's HQ — you don't have clearance. So you make your own copy of it, do your work there, and offer it back. This is how nearly everyone makes their first open-source contribution.

① Fork it. On their GitHub repo, click Fork (top-right). You now have an identical copy under your own account.

② Clone your fork and work on a branch exactly like Operation II:

terminal · note: YOUR username, not theirs
$ git clone git@github.com:YOUR-NAME/their-repo.git
$ cd their-repo
$ git switch -c fix/typo-in-readme
# make your change, then:
$ git add . && git commit -m "Fix typo in README"
$ git push -u origin fix/typo-in-readme

③ Open the Pull Request — but this time point it from your fork into the original repo. GitHub offers this automatically after you push. The project's maintainers review it and, if they like it, merge it. 🎉

That's the whole open-source loop: fork → branch → commit → push → PR to the original. You've now got every core move in the trade.

Code Red — a secret got out
The Leak strikes

Caught before the push left the building? Break the seal, remove the secret, re-seal:

terminal
$ git reset --soft HEAD~1
# unseals the last crate but keeps your files — now scrub the secret,
# add it to .gitignore, and commit again

Already delivered to HQ? Assume the enemy has read it. Revoke and rotate that credential immediately — a key you've replaced is worthless to them. Then scrub history with git filter-repo or BFG. Rotating beats scrubbing, always: you can't un-see what's been seen, but you can make it useless.

Before you go live

Rules of engagement — working on shared ground

This manual taught you the moves. Production adds one layer more: other agents are standing on the same ground as you. Break something here and you block the whole team. These habits keep everyone safe.

① Ask HQ before you act

Every agency runs differently, and no manual can guess your team's rules. On day one of a real operation, get these answered before you touch anything shared:

  • What's the branch naming convention? (e.g. feature/…, fix/…)
  • Does the team merge or rebase — and who is allowed to click Merge?
  • Which branches are protected? (Usually you can't push to main directly — that's a good thing.)
  • How many reviewers must approve a PR? Are there required checks or CI that must pass?
  • Is there a commit message format the team expects?

② Never do these on a shared repo

✗
Never push straight to main. Always branch → PR → review → merge. No exceptions on shared ground.
✗
Never force-push a shared branch (git push --force). It rewrites history other people rely on and can erase their work. If you ever think you need it — stop and ask first. There's almost always a safer route.
✗
Never resolve a conflict by blindly deleting the other side. Read both versions, understand what each does, and keep what's correct. A careless conflict fix silently deletes a teammate's code — one of the most common real-world incidents.
✗
Never commit a secret — not even once, not even briefly. On a shared repo, assume every person and bot sees it the instant you push. If it happens, rotate the credential immediately (see Code Red).
✗
Never start work on a stale copy. Run git switch main && git pull before you branch, so you build on the latest — not last week's.

③ When something breaks — don't dig deeper

Mistakes happen to everyone, seniors included. The pros aren't the ones who never break things — they're the ones who stop and ask before making it worse. Git almost never truly loses work: git reflog can recover "lost" commits, and a bad merge can be reverted. If you're unsure, freeze and ask a teammate rather than firing off more commands in a panic.

Your first weeks, played safe

Work only on branches, open PRs, and let someone review your merges. Do that and you effectively can't break production. Speed and confidence come with reps — the safety habits come first.
